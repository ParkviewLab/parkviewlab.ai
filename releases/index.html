jonobones v0.1.4

A headless, Joplin-sync-compatible knowledge daemon.

Released 2026-06-12 AGPL-3.0-or-later npm: jonobones · @parkviewlab/jonobones container image
What this version provides
Owns a private Joplin-format knowledge base (notes, notebooks, tags, resources) and keeps it synchronized with any Joplin sync target — Joplin Server/Cloud, filesystem, WebDAV/Nextcloud, Dropbox, OneDrive, or S3 — serving it to local applications over a localhost /v1 REST API. Clients speak HTTP only; the SQLite database, resource files, sync metadata, and encryption keys stay private to the daemon. CLI control plane, token auth on every endpoint except /health, and service install for launchd (macOS) / systemd (Linux). Ships on npm and as a container image.
Install
npm install -g jonobones
# or
docker pull ghcr.io/parkviewlab/jonobones:latest

deco-assaying v0.2.1

MCP server for source-code analysis via tree-sitter (Python/C/C++ initially, broader coverage planned).

Released 2026-06-13 MIT PyPI: deco-assaying container image
What this version provides
Feeds structural information about a codebase — symbols, imports, references, cAST chunks, and metrics — into downstream consumers. Async index_repo jobs accept a local directory, a GitHub URL, or a GitLab URL (including nested groups); results are retrievable over MCP tools or an HTTP download API (manifest, analysis index, per-file artifacts). Partial-clone, bin-packed batched fetching keeps the source-side disk footprint at ~100 MB regardless of repo size. Ships MCP prompts that encode the recommended workflow, plus optional GITHUB_TOKEN/GITLAB_TOKEN for higher quotas and private repos.
Install
uvx deco-assaying
# or
docker pull ghcr.io/parkviewlab/deco-assaying:latest

flint-slating v0.1.4

MCP server that reads PDFs and exposes them as structured Markdown.

Released 2026-05-18 MIT PyPI: flint-slating container image
What this version provides
Converts PDFs to LLM-ready Markdown with heading hierarchy, multi-column reading order, and Markdown tables, plus ancillaries: metadata, outline, images, and per-page tables. Built entirely on a permissive-license stack (Docling, pypdf, pdfplumber) — no PyMuPDF, no AGPL/GPL in the dependency tree, enforced by a CI license-check. Runs as a Streamable-HTTP daemon or a stdio MCP server; large PDFs queue a background job (hybrid sync/async), and torch is pinned to CPU-only wheels so no CUDA libraries are bundled.
Install
uvx flint-slating
# or
docker pull ghcr.io/parkviewlab/flint-slating:latest

smalt-mcp v1.2.0

MCP wrapper around Smalt's storage surface (read / write / link / claim / search).

Released 2026-05-18 MIT PyPI: smalt-mcp container image
What this version provides
Per the project's status: the storage substrate is complete — 17 tools across three permission tiers (read_only / read_write / remove_destructive), with an auto-indexer trigger on writes and hybrid search (FTS + vector + alias, RRF-fused) over a markdown + LanceDB store. A thin, single-writer wrapper with no agentic logic, serving as the storage layer for ParkviewLab's CoGrind project. Streamable-HTTP transport with health, admin, and OpenAPI endpoints.
Install
uvx smalt-mcp
# or
docker pull ghcr.io/parkviewlab/smalt-mcp:latest

ebony-enriching v0.1.4

An MCP lab notebook — proposal / experiment / gap lifecycle.

Released 2026-05-17 MIT PyPI: ebony-enriching container image
What this version provides
Per the project's status: the v0.1 surface is complete — 13 tools across two permission tiers (6 read-only, 7 read-write) covering the full proposal / experiment / gap lifecycle. Append-only lab-notebook semantics (proposals transition status rather than delete; experiments are the historical record); the server enforces storage correctness only, leaving lifecycle policy to the agents using it. Five operating modes — uvx, uv tool install, launchd, systemd, and Docker/GHCR.
Install
uvx ebony-enriching
# or
docker pull ghcr.io/parkviewlab/ebony-enriching:latest
About the summaries These projects publish git tags and packages, but do not currently publish GitHub Releases or maintain CHANGELOG files, so there are no formal per-version release notes to quote. The “What this version provides” text summarizes each project's own README (including its “Status” section where present) and describes the current shipped surface — not a diff of what changed in this specific version. To put true per-release notes on this page, publish a GitHub Release (or keep a CHANGELOG) for each tag and the notes can be pulled in automatically.
